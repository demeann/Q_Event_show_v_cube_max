# ТЗ (нормализованная версия) — Telegram-бот Q CLUB

## 1) Статус документа
- Этот файл является канонической версией требований для MVP Telegram-бота.
- Исходные файлы `Q-CLUB.md`, `tz.md`, `additonal_conditione.md` считаются источниками контекста, но при конфликте приоритет у этого файла.

## 2) Цель проекта
- Запустить игровую механику "Шоу в кубе" в Telegram-боте.
- Механика состоит из 3 туров (по мотивам ТВ-шоу), каждый тур длится 3 дня.
- По итогам каждого тура определяется 9 победителей.
- После этапа Telegram-бота планируется зеркало в MAX (вне текущего MVP).

## 3) Границы MVP
- Канал: Telegram-бот.
- Обязательные модули:
  - пользовательский сценарий прохождения 3 туров;
  - учет ответов и баллов;
  - выбор победителей по алгоритму "считалочки" (Иосиф Флавий);
  - механизм массовых рассылок;
  - админ-функции просмотра результатов и выгрузок CSV/Excel.
- Не входит в MVP:
  - ручной старт/стоп туров из админки;
  - ручное продление, ручной пересчет, ручное открытие/закрытие вопросов.

## 4) Обязательные бизнес-правила
- Туров: 3.
- Длительность каждого тура: строго 3 дня (без ручных сдвигов в MVP).
- Победителей в каждом туре: 9.
- Количество попыток на вопрос: 1.
- При неверном ответе баллы не отнимаются.
- Время событий фиксируется по МСК.
- `timestamp` последнего ответа в туре сохраняется для аналитики активности, но не используется как tie-break при выборе победителей.

## 5) Валидация пользователя перед стартом
- Перед участием требуется подтверждение email.
- Разрешенные домены:
  - `@pmru.com`
  - `@contracted.pmru.com`
- При невалидном домене доступ к механике не открывается.

## 6) Структура туров

### Тур 1 (дни 1-3): "Кто хочет стать миллионером"
- Формат: 5 вопросов.
- Каждый вопрос: 4 варианта ответа, 1 правильный.
- За правильный ответ начисляются баллы.
- За неверный ответ: показать правильный вариант, баллы не уменьшать.

### Тур 2 (дни 4-6): "Своя игра"
- Формат: 3 темы, в каждой теме по 2 вопроса (итого 6 вопросов).
- Пользователь выбирает тему.
- Внутри выбранной темы:
  - если ответ на 1-й вопрос верный, пользователь получает доступ ко 2-му вопросу темы;
  - если ответ неверный на 1-м или 2-м вопросе, тема завершается для пользователя;
  - баллы не отнимаются.
- После завершения темы пользователь может перейти к следующей теме, пока не завершит доступные темы тура.

### Тур 3 (дни 7-9): "Где логика"
- Формат: 5 вопросов (зафиксировано).
- Каждый вопрос: картинка-шифровка + 2 варианта ответа, 1 правильный.
- За правильный ответ начисляются баллы.
- За неверный ответ: показать правильный вариант, баллы не уменьшать.

## 7) Массовые рассылки (обязательно в MVP)
- Нужен механизм массовых рассылок в Telegram.
- Типы рассылок:
  - анонс тура;
  - напоминание о текущем туре;
  - сообщение о завершении тура / анонс следующего.
- Минимально обязательный сценарий: по одному анонсу на каждый из 3 туров.
- Дополнительно поддержать отправку напоминаний по сегментам (например, "не начинали тур", "начали, но не завершили").
- Рассылки должны логироваться (статус отправки, время, аудитория/сегмент, текст шаблона).

## 8) Алгоритм выбора 9 победителей (обязательно)

### 8.1 Пул кандидатов
- После завершения тура формируется список кандидатов из участников с максимальным баллом тура.
- Если кандидатов меньше 9, требуется отдельное бизнес-правило (по умолчанию: добор из следующего уровня баллов в порядке убывания балла, затем применение того же алгоритма).

### 8.2 Базовый порядок списка
- Кандидаты упорядочиваются детерминированно и без использования времени ответа.
- Рекомендуемый порядок для MVP: по `user_id` (или `telegram_user_id`) по возрастанию.

### 8.3 Выбор победителей
- Используется алгоритм "считалочки" (аналог задачи Иосифа Флавия):
  1. Есть упорядоченный список `L` длиной `T`.
  2. Нужно выбрать `W = 9` уникальных победителей.
  3. Шаг `n` считается от текущей позиции по циклу.
  4. Выбранный участник удаляется из `L`.
  5. Следующий отсчет начинается с участника, следующего за удаленным.
  6. Повторять, пока не выбрано 9 победителей.
- Значение `n` для тура:
  - по умолчанию рассчитывать автоматически: `n = ceil(T / W)`, минимум 1;
  - сохранить `n` в результатах тура для прозрачности и аудита.

## 9) Админ-функции MVP
- Просмотр статистики тура/пользователей.
- Просмотр результатов и списка победителей.
- Выгрузка отчетов по кнопке:
  - CSV;
  - Excel.
- Ручные кнопки управления турами в MVP не требуются.

## 10) Данные и аудит
- Хранить:
  - профиль участника;
  - подтвержденный email;
  - ответы по вопросам;
  - набранные баллы;
  - время ответов (МСК);
  - факт и параметры рассылок;
  - результаты тура и параметры выбора победителей (`T`, `W`, `n`, порядок списка).
- Все ключевые действия должны быть воспроизводимыми по логам.

## 11) UX/тексты
- Стиль коммуникации: как в исходном ТЗ (легкий, дружелюбный, с фирменной тональностью Q CLUB).
- Вопросы/варианты из исходных материалов использовать как контентную базу для MVP.

## 12) Нормализованные решения по конфликтам исходных документов
- Тур 1: фиксировано 5 вопросов (вместо "3-5").
- Тур 3: фиксировано 5 вопросов (вместо "3-5" и расширенных таблиц на 7 позиций).
- Победители: не tie-break по времени, а "считалочка" (Иосиф Флавий) с удалением выбранных.
- Время ответа: сохраняется только как аналитический показатель.
- Пуши:
  - обязательные анонсы по 1 на тур;
  - плюс поддержка дополнительных массовых рассылок/напоминаний через механизм рассылок.
- Строгие 3 дня на тур, без ручных сдвигов в MVP.
